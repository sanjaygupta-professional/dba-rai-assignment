Explainability, in three lenses.

Explainability isn't one technique — it's a question of what you're explaining: the data a model learned from, the model's internal logic, or a specific output it produced. Each lens answers a different stakeholder's question, and answering the wrong one is the most common way an "explainable" system still fails an audit.

Why this matters beyond the lecture

DARWIN's R = Responsibility pillar treats explainability as a product requirement, not a compliance afterthought: a system that can't say why loses the human's trust the first time it's wrong. In CareBridge, the reviewer packet doesn't compute a black-box score — every recommendation is a chain of cited, checkable claims. That design choice is explainability; this page is the vocabulary for arguing why it counts.

The taxonomy

Three lenses, mapped to where in the ML lifecycle they operate. Data-based looks before the model trains; model-based looks inside it; results-based looks at what came out, for one case or many.

A second, orthogonal split cuts across all three: global explanations describe the model's behavior on average, across every input (a regulator's question); local explanations describe this one prediction (a reviewer's question, and the one CareBridge answers). Most named techniques — SHAP, LIME, counterfactuals — pick a side of this axis; know which one before you reach for it.
Lens 1 · pre-model

Data-based explainability

Before asking why the model decided something, ask whether its training data could ever have supported a fair, correct answer. Most real-world bias and failure enters here, not in the algorithm.

TechniqueWhat it showsTypical use
Datasheets / dataset documentationProvenance, collection method, intended use, known gaps — a structured "nutrition label" for the dataset.Governance review, vendor/model-card handoff.
Exploratory data analysis & visualizationDistributions, correlations, outliers — surfaces skew before training starts.First pass on any new dataset.
Data lineage / provenance tracingWhere each field came from and every transform applied to it.Debugging a metric that looks wrong downstream; audit trails.
Missingness & quality profiling% missing per field, imputation risk — a field missing >15–20% is usually unsafe to impute for clinical/financial evidence.Go/no-go gate before training.
Data-leakage checksWhether a feature is only available because the outcome already happened (label leakage) or the training view doesn't match the inference-time view.The single highest-value check before trusting any accuracy number.
Subgroup / slice analysisWhether performance and representation hold up within slices (age band, accent, dialect, facility type) not just in aggregate.Fairness and training-bias review.
Dimensionality-reduction views (PCA, t-SNE, UMAP)Whether classes/clusters separate cleanly in the input space, or overlap in ways that predict future confusion.Diagnosing a high-error class before blaming the model.
Recency / drift checksWhether the training window still reflects current practice (coding updates, policy versions, seasonal shift).Deciding retraining cadence.
CareBridge example. data-document.md runs exactly this lens: missing-% ceilings, a label-leakage check (scoring against historical outcomes the model would never see at inference), training-bias review by facility/payer mix, and a recency check on policy versions. This is the lens most course rubrics grade under "Data Document."
Lens 2 · in-model

Model-based explainability

Two forks decide which technique applies: is the model intrinsically interpretable or a black box needing a post-hoc method bolted on — and do you need to explain the model globally or one prediction locally?

Intrinsic / ante-hocPost-hoc, model-agnosticPost-hoc, model-specific
TechniqueFamilyScopeWhat it shows
Linear/logistic regression, decision trees, rule lists, GAMsIntrinsicGlobalThe model is its own explanation — coefficients or splits are the reasoning.
Permutation feature importanceModel-agnosticGlobalWhich features move accuracy most when shuffled — a coarse "what matters overall."
Partial Dependence / ICE plotsModel-agnosticGlobal (ICE: per-instance)How the prediction changes as one feature varies, holding others fixed.
Surrogate modelsModel-agnosticGlobal or localAn interpretable model (e.g. a shallow tree) trained to mimic the black box's outputs — explains the mimic, not the original, so fidelity must be checked.
SHAP (Shapley values)Model-agnosticBothEach feature's contribution to one prediction, game-theoretically fair and additive; aggregates cleanly to a global view. Expensive; most rigorous.
LIMEModel-agnosticLocalFits a simple interpretable model in the local neighborhood of one prediction. Cheap, intuitive, but unstable — small input changes can flip the explanation.
AnchorsModel-agnosticLocalA minimal if-then rule that "anchors" a prediction with high precision — easier for a non-technical reviewer to read than a SHAP plot.
Saliency maps / Grad-CAMModel-specific (vision/deep nets)LocalWhich pixels or regions most influenced an image prediction.
Attention-weight visualizationModel-specific (transformers)LocalWhich tokens the model attended to — suggestive, not proof of causal influence.
Integrated gradientsModel-specific (deep nets)LocalAttributes the prediction to input features via a gradient path from a baseline — more theoretically grounded than raw saliency.
Caution built into the lens itself. Post-hoc explanations describe the explainer's approximation of the model, not the model's true internal process — fidelity (does the explanation match what the model actually did?) and stability (does a tiny input change flip the explanation?) are both real failure modes, especially for LIME. This is why CareBridge sidesteps this lens for the packet itself: it uses a structured extraction pipeline (evidence → policy criterion, cited) instead of an opaque score plus a post-hoc excuse for it.
Lens 3 · post-model

Results-based explainability

Sometimes the stakeholder doesn't want the mechanism at all — they want to know whether this specific answer can be trusted, and what would have to change to get a different one.

TechniqueWhat it showsTypical use
Counterfactual explanationsThe minimal change to the input that would flip the outcome ("if income were $4K higher, this would have been approved").Actionable explanations for an end user, adverse-action notices (ECOA-style).
Example-based (prototypes & criticisms)The closest training examples that support the prediction, and the ones that most contradict it.Building trust with a domain expert who reasons by case comparison.
Confidence & uncertainty quantificationHow sure the model is, not just what it predicted — a calibrated probability or a prediction interval.Deciding when to route to a human instead of trusting the output.
Calibration checks (reliability diagrams, ECE)Whether stated confidence matches empirical accuracy (a model "95% confident" that's right only 70% of the time is miscalibrated).Validating a groundedness or confidence gate before it's trusted as a hard stop.
Rationale / natural-language justification generationA generated explanation in plain language, ideally grounded in cited source spans rather than a free-floating narrative.Reviewer-facing packets, LLM-based decision support.
Error / failure-mode analysisWhere and how the model is systematically wrong — clustering failures by subgroup, input type, or confidence band.Post-deployment monitoring, red-teaming before launch.
Audit trail / source citationEvery assertion traceable to the exact evidence span it came from — the explanation isn't generated after the fact, it's the mechanism itself.Regulated decision support where "trust me" isn't acceptable.
The distinction that matters in Q&A. A model-based explanation (SHAP) says "feature X contributed +0.3 to this score." A results-based one says "here's the cited evidence for this recommendation, and here's what's missing." The second is what a clinician, underwriter, or auditor actually acts on — which is why "explainability" in a product sense usually means results-based, even when the technique underneath is model-based.

Choosing a technique

Start from the stakeholder's question, not from the technique you already know.

Stakeholder asks…LensReach for
"Could the training data itself be unfair or leaking?"Data-basedSlice analysis, leakage check, datasheet
"What does the model weigh most, in general?"Model-based, globalPermutation importance, PDP, or an intrinsic model
"Why did it flag this case?"Model-based, localSHAP or LIME (tabular); saliency/Grad-CAM (image); attention (text)
"What would change the outcome?"Results-basedCounterfactual explanation
"How sure are you, and should a human check this?"Results-basedCalibrated confidence + a routing threshold
"Prove it, with evidence I can check."Results-basedCited rationale / audit trail
Two constraints prune the table fast: model type (intrinsic models don't need a post-hoc method at all) and data modality (saliency/Grad-CAM are vision-specific; SHAP/LIME/PDP assume tabular or structured features; attention and integrated gradients fit text/sequence models).

Applied — CareBridge, all three lenses at once

The reviewer packet is graded, in part, on whether it demonstrates explainability discipline — here's how each lens shows up in the build already on this site.

D

Data-based

usecases/healthcare/data-document.md — missing-% ceiling, label-leakage check (scoring history vs. inference-time mirror), training-bias review by facility/payer, recency check on policy versions.

M

Model-based

Deliberately avoided as the primary trust mechanism — a structured evidence-to-criterion extraction pipeline is closer to an intrinsic, rule-checkable model than a black-box score needing a post-hoc excuse.

R

Results-based

Every packet assertion carries a source citation (note ID + span) and a met/unmet/uncertain status; groundedness ≥95% is a calibrated hard-stop gate, not a vibe. See DARWIN-R in the full report.

If Q&A asks "how would you explain this to an auditor?" — answer with the lens, not just the technique: "Results-based — every claim traces to a cited source span, and the confidence gate is calibrated, not asserted."

Glossary

Ante-hoc / intrinsic
Interpretable by design (linear model, shallow tree) — no separate explanation step needed.
Post-hoc
An explanation generated after training, for a model that isn't interpretable on its own (SHAP, LIME, saliency).
Global explanation
Describes model behavior across the whole input space — "on average, what matters."
Local explanation
Describes one specific prediction — "why this case, this time."
Fidelity
How well a post-hoc explanation actually matches what the model did internally.
Stability
Whether a small input change produces a wildly different explanation (a known LIME weakness).
Feature attribution
Assigning a share of the prediction's "credit" to each input feature (SHAP, integrated gradients).
Counterfactual
The minimal input change that flips the model's output.
Calibration
Whether a model's stated confidence matches its real-world accuracy at that confidence level.
Data leakage
A feature that's only available because the outcome already happened, or a training view that doesn't match inference-time inputs.
Groundedness
Whether a generated claim is actually supported by the cited source, not fabricated.
